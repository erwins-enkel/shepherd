# Token-usage analysis — five recent authoring tasks

This report gives a per-session token breakdown plus a sustainability read of **five recent _authoring_ task-sessions** (TASK-286/287/288/290/295). It is the factual basis for a **later** optimization pass — it lists optimization levers as proposals only, it does not apply them. Operational archetypes (merge-trains, `/impeccable` audits) are excluded from the sample and quantified separately in the appendix. All figures are generated by [`scripts/usage-report.ts`](../scripts/usage-report.ts) — re-run it (`bun run scripts/usage-report.ts --md`, or `--include-excluded` for the appendix) after any optimization to measure the delta.

## Methodology

How cost is attributed per task:

- **Authoring** — the pinned-session transcript (the task's `claudeSessionId` JSONL in its worktree project dir). This is the task's own work and the basis for the authoring columns.
- **In-worktree ancillary** (`Anc`) — every _other_ `*.jsonl` in the pinned session's worktree project dir, reported **separately** (never folded into authoring) and classified `probe` / `resumed` / `unknown`: a tiny model-detection probe, a resumed/longer transcript, or anything else.
- **Satellite passes** (`Sat`) — the critic/PR-review, plan-gate, and plan-review sessions Shepherd spawns per task, attributed via a **DB-first** chain: a `reviews` / `plan_gates` row is authoritative → else a plan-review dir links by its **embedded task-session UUID** → else a critic dir links by **transcript content-match** (each critic transcript opens with the reviewer preamble and embeds the task's `prompt` + `baseBranch`), optionally **sha cross-checked** against the repo's commits. Each linkage carries a tag: `db`, `content`, or `sha-confirmed`. A task can own **multiple** critic dirs (the critic re-runs per reviewed head) — all summed. Review dirs that match no task but whose activity falls inside a task's time window are flagged as an **unlinked-in-window residual**, never merged into a task. The account-wide periodic distiller is out of scope (not per-task).

**Honest caveats — read before trusting any number:**

- For this sample the `reviews` and `plan_gates` tables (and the spawn log) were **empty**, so no satellite linked via `db`. **Content-match is the operative link** here — every `Sat tags` value is `content` and/or `sha-confirmed`, reconstructed after the fact, not recorded at spawn.
- The cost figure (`cost*`) is a **relative per-Mtok proxy**, never dollars. It is `src/pricing.ts`'s `weightedUnits` — input/output weights Opus 5/25, Sonnet 3/15, Haiku 1/5, Fable 10/50, with cache kinds derived at fixed 0.1× / 1.25× / 2× ratios. **Only ratios between rows are meaningful**; the absolute scale is arbitrary.
- The sample is **5 tasks** — directional, not statistically robust. Treat the patterns as signals, not measurements.

## Sustainability analysis

The token bill of an authoring session is **not** I/O of fresh code or model output — it is the agent re-reading its accumulated context, over and over, every turn. The data makes that concrete:

- **Cache-reads dominate: 91–99% of every task's token volume is `cacheRead`.** Fresh input is tiny (16–24K tokens/task); output is a sliver (4.5K–170K). The overwhelming majority of tokens is the agent re-reading its accumulated context on every message. The lever is therefore **not** I/O — it's **context size × turn count**. Anything sitting near the top of context — the system prompt, `CLAUDE.md`, injected `<system-reminder>` blocks, house-rules, bloated tool results — is re-read on _every_ one of the session's messages.
- **Cost scales steeply with session length / turn count.** TASK-288 (157 msgs, 3h10m) cost 37.4M tokens; TASK-295 (13 msgs, 6m) cost 0.66M — a ~57× spread. `cacheRead` compounds with _both_ a growing context window and more turns over it, so long sessions pay super-linearly.
- **Cache writes are a real cost, but the old `cacheWrite/input` "churn" ratio was a denominator artifact (investigated in #500).** Claude Code caches the whole context every turn, so fresh `input` is tiny and opener-dominated — for TASK-286, 15,478 of its 19,371 input tokens (~80%) is request #0 alone; the rest average ~59/turn. Dividing the steadily-growing cache writes by that near-zero denominator inflated the figure to "27×"; it never meant the prefix was rebuilt 27 times. Decomposed per request, `cacheWrite` is dominated by unavoidable incremental cache extension plus occasional one-off large-content writes; genuine _full_ prefix rebuilds (`cacheRead` warm→cold) happen at most once per session — the new **`FullRecache`** column. What does matter is the _cost_ weight: by the weighted proxy, `cacheWrite` is **18–34% of authoring cost-units** for the long Opus tasks — the new **`CacheWcost%`** column — far above its ~3% token share, because write weights are 1.25×/2×. The lone full rebuilds show no Shepherd-injected mutating prefix block: TASK-286's rebuild had a 7-second gap (not the 5-min cache TTL), same process and permission mode, an already-stable branch, and no context shrink (so not auto-compaction). It is consistent with Anthropic-side cache eviction — a resume/herdr-reattach respawn or compaction can't be excluded in general — but in every case there is **no frequently-mutated near-top block to stabilize**.
- **Review/critique overhead is real but secondary — and under-counted.** Linked satellite cost is **0.25× authoring** by the weighted proxy (≈11% by raw tokens: 7.9M sat vs. 69.4M auth). But the **unlinked-in-window residual** (8 dirs, 5.6M tokens, 8.3 cost-units) is roughly _half the linked satellite cost again_ — some of it plausibly belongs to these tasks but couldn't be confidently linked (content-match only, no DB rows), so true review burn is likely **above** 0.25×. The critic also re-runs per reviewed head in the auto-address loop: **TASK-288 spawned 5 critic dirs.**
- **Premium model tiers cost ~2× per token.** The Fable task (TASK-290) shows a higher cost-proxy _per token_ than the Opus tasks — Fable's weights are double Opus's. Using a premium tier for a content/audit pass is a deliberate multiplier, not free.
- **Probes are negligible.** TASK-286's two in-worktree probe sessions total 84K tokens — a rounding error against the 69.4M authoring total.

Net: the dominant cost driver is **per-turn context re-read**, amplified by session length. Review overhead is the clear second-order cost, and model-tier choice is a third multiplier on top.

## Optimization levers (proposals — not done here)

Ranked by expected leverage, each tied to evidence above. These are proposals for a **later** pass; nothing here is implemented.

1. **Trim per-turn context bloat (highest leverage).** Because cost is `cacheRead × every message`, every KB removed from the always-present preamble is saved on _every turn of every session_. Audit the system prompt, `CLAUDE.md`, repeated `<system-reminder>` / house-rules injections, and verbose tool outputs. _Evidence:_ 91–99% of every task is `cacheRead`.
2. **Cache-write churn — investigated (#500), no stabilization needed.** The `cacheWrite/input` "churn" ratio was a denominator artifact, not prefix invalidation; no frequently-mutated near-top block exists. The misleading `CacheWchurn` column was replaced with **`FullRecache`** (genuine full-rebuild count, ≤1/session) + **`CacheWcost%`** (cache-write cost share). `cacheWrite`'s cost (~18–34% for long Opus tasks) is dominated by unavoidable incremental writes — reduced only by lever #1 (smaller per-turn context → smaller writes), not by stabilizing any block. _Evidence:_ per-request decomposition of TASK-286/288/290 (see Sustainability analysis).
3. **Bound critic re-runs / dedupe reviews.** Confirm the patch-id rebase-skip (`review.ts` `rebaseSkip`) actually suppresses redundant re-reviews, and consider a per-task cap on auto-address rounds. _Evidence:_ TASK-288 spawned 5 critic dirs; review burn is the clear second-order cost and is likely under-counted (residual ≈ half the linked satellite cost).
4. **Right-size model tier per task kind.** Reserve premium tiers (Fable) for work that genuinely needs them; default mechanical tasks to a cheaper tier. _Evidence:_ Fable weights are ~2× Opus per token (TASK-290 / 291 / 292).
5. **Shrink the critic's own context.** Each critic round opens a fresh session re-reading the full diff + task prompt + prior findings; trim what it's fed. _Evidence:_ satellite passes carry their own (sometimes large) `Sat tok` totals.

**Meta-lever (closes the measurement gap):** persist review / plan-gate session-ids + token totals **at spawn**, so future attribution is exact. The unlinked-in-window residual exists precisely because linkage is reconstructed after the fact (content-match) rather than recorded — fixing that makes every later report's review-cost number trustworthy instead of a lower bound.
