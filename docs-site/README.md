# `docs.shepherd.run` documentation site

The documentation site for **Shepherd**, served at `docs.shepherd.run`. Static
[Astro](https://astro.build) + [Starlight](https://starlight.astro.build) site —
no backend, no secrets.

Built across the docs epic (#875): brand-themed off the UI tokens, with curated
guide pages plus four generators that keep the reference sections in sync with
the repo — the TypeScript API reference (TypeDoc over `../src`), the imported
`reference/` markdown (`docs/*.md` + `CLAUDE.md`), the operator `herdr` CLI
reference, and the `llms.txt` family. See **Generated content** below.

## Why `docs-site/` (not `docs/`)

`docs/` is an **active content-source directory** with live write conventions —
`RESEARCH_DIRECTIVE` writes `docs/research/<slug>.md` and the brainstorming flow
writes `docs/superpowers/specs/` — so the Starlight package can't root there.
It lives at the sibling path `docs-site/` (parallel to `site/`), and Vercel's
**Root Directory** is set to `docs-site`.

## Relationship to the app's gates

Like `site/`, this package is **not** part of the Shepherd app and is exempt from
the app's i18n (EN/DE parity), glossary, and feature-announcement-catalog gates.
Those gates are scoped to `ui/` + `extension/` and to `feat(...)` commits touching
`ui/src/...`, so `docs-site/` never trips them. `docs-site/` is added to the root
`.prettierignore` (its `.astro`/`.md` files use Astro's own toolchain) and to the
root `tsconfig.json` `exclude` (it type-checks via `astro check`, not the server
`tsc`).

Unlike `site/`, this package **stays in the monorepo**. Both are wired into CI
(`.github/workflows/ci.yml`), each in its own job: `cd docs-site && bun install
--frozen-lockfile && bun run check && bun run build`.

## Generated content

Most of the site is **generated from the repo**, not authored here. Three of the
four generators write **git-ignored, build-time** paths under `src/content/docs/`
(or `dist/`) so they cannot drift from their source; the fourth (the CLI
reference) is **committed** and kept fresh by a regenerate-on-merge gate (below).

- **`reference/`** — in-repo markdown imported verbatim by `scripts/sync-docs.mjs`
  (called from `astro.config.mjs`). Sources of truth: `docs/external-task-api.md`,
  `docs/sandbox-security.md`, and the repo-root **`CLAUDE.md`** (rendered as the
  **Project house rules** page). No re-authoring — each is stripped of its body
  `# H1`, given frontmatter, and has its repo-relative links rewritten to GitHub.
- **`llms.txt`** — `llms.txt` / `llms-full.txt` / `llms-small.txt`
  ([llmstxt.org](https://llmstxt.org)) for LLM/agent consumption, generated into
  `dist/` by [`starlight-llms-txt`](https://github.com/delucis/starlight-llms-txt)
  during `astro build` (served at `/llms.txt` etc.). The large TypeDoc API
  reference is `demote`d to the end of every output and `exclude`d from the small
  output (`llms-small.txt`) so that subset stays lean.
- **`api/`** — the **TypeScript API reference**, generated by
  [`starlight-typedoc`](https://github.com/HiDeoo/starlight-typedoc) (TypeDoc +
  typedoc-plugin-markdown) from the server package `../src` using the root
  `../tsconfig.json`. Source of truth: `src/`. It documents the whole tree
  (`entryPointStrategy: "expand"`; the lone test file + the zero-export
  `src/index.ts` entry script are excluded), so the reference can't drift from
  the code. The autogenerated sidebar group (`typeDocSidebarGroup`) is appended
  to the Starlight `sidebar` as the **API reference** section.

  Output is **reproducible from `src/` alone**: `disableGit` + a pinned
  `sourceLinkTemplate` keep "Defined in" links repo-relative and free of any
  commit SHA or machine-absolute path, so two builds at the same source state are
  byte-identical. Because it runs from `astro.config.mjs` at init time, every
  `astro` command (`check` / `build` / `dev`) regenerates it — no separate script
  or CI step.

- **`reference/cli/`** — the operator-facing **herdr CLI reference**, generated by
  `scripts/gen-cli-reference.ts` from live `herdr --help` and **committed** (run
  `bun run gen:cli`). Unlike the three above, `herdr` is absent from CI/Vercel, so
  these pages cannot be built there — they are a point-in-time snapshot pinned to
  one `herdr` release. Freshness is enforced by the gate below.

## Keeping committed generated output fresh

`reference/cli/` is the only **committed** generated artifact, so it is the only one
that can drift. `scripts/check-generated-docs.sh` (a [PR-hygiene](../.github/workflows/pr-hygiene.yml)
+ pre-push gate, mirroring `check-feature-catalog.sh`) **fails** a PR that changes a
generator source (`gen-cli-reference.ts`) without regenerating its committed output —
fix with `bun run gen:cli` and commit, or opt out with `[skip-docs-regen]` when a
source edit leaves the output byte-identical. It is a **presence heuristic**, not a
byte-level freshness check (CI has no `herdr`); the other three generators need no
gate because every `astro build` regenerates them into git-ignored paths.

## Develop

```bash
cd docs-site
bun install
bun run dev      # local dev server
bun run build    # static output to dist/
bun run preview  # serve the built dist/
bun run check    # astro check (type + content diagnostics)
```

## Brand tokens

`src/styles/custom.css` maps a curated subset of the UI design tokens
(`ui/src/app.css` `--color-*` / `--fs-*`) onto Starlight's `--sl-*` variables. The
literal values live in a single `--brand-*` provenance block (mirroring
`app.css`); every `--sl-*` mapping references those via `var()`, so no hex is
scattered through the theme — the same pattern as `site/src/styles/global.css`.

## Go-live (manual operator steps — creating from scratch)

The build is **deployment-inert** until an operator creates and wires a Vercel
project. Nothing is served at `docs.shepherd.run` until these steps are done.

1. **Create the Vercel project (operator-gated — not done autonomously by an
   agent).** In Vercel: **Add New → Project**, import the `shepherd` monorepo, and
   set **Root Directory = `docs-site`**. Astro is auto-detected (build
   `astro build`, output `dist/`); the shipped `vercel.json` pins
   `"framework": "astro"`.
2. **Attach the domain:** Project → **Settings → Domains** → add
   `docs.shepherd.run`, then create the **DNS records Vercel prescribes** (a
   `CNAME` for the `docs` subdomain) at the `shepherd.run` registrar. The root
   `shepherd.run` is already live on its own (marketing) project; this adds the
   `docs` subdomain alongside it.
3. **Deploy**, then verify:
   ```bash
   curl -fsSL https://docs.shepherd.run/ | head   # serves the docs landing HTML
   ```

Until the domain is attached and DNS propagates, the site is reachable only at
the Vercel-assigned preview URL.
